## 快速开始（新成员）
1. 先阅读 [`目录.md`](./目录.md) 了解章节边界，再按 `01 -> 09` 顺序阅读正文。
2. 确认本次修改落在哪个章节，避免跨章节重复写同一规则。
3. 修改文档后，必须同步更新 [`版本更新内容.md`](./版本更新内容.md) 置顶条目。
4. 提交代码时使用规范前缀：`docs:` / `feat:` / `chore:`。
5. 推送前检查远程与分支，默认推送到 `origin/main`。

## 核心附件与用途
- [`Reference/`](./Reference/)：制度依据与研究输入（依据层）。
- [`01-09` 章节文档](./目录.md)：规范正文与可执行方案（输出层）。
- [`附件-国内外相关政策法规参考材料.xlsx`](./附件-国内外相关政策法规参考材料.xlsx)：法规映射与对照底稿，用于：

  - 评审会中的引用与快速比对。
  - 章节修订时的国内/国际口径对齐。

## 协作与提交流程
最小闭环流程：`本地修改 -> 更新版本更新内容 -> git add/commit -> git push origin main`

协作规则：
1. 每次文档修改后，必须在 [`版本更新内容.md`](./版本更新内容.md) 追加置顶记录。
2. 提交信息必须带类型前缀：`docs:` / `feat:` / `chore:`。
3. 需要推送 GitHub 时，默认目标为 `origin/main`（当前已验证可用）。
4. 变更总结应说明“修改前 -> 修改后 -> 修改意义”，禁止只写笼统描述。

## 编制原则
1. **制度先行**：以授权规则与管理办法作为系统设计约束。
2. **闭环执行**：申请、审批、签约、支付、交付、留痕、复核全链路闭环。
3. **安全优先**：数字资源全生命周期安全与可追溯优先于便利性。
4. **可运营**：规则可配置、价格可计算、流程可审计、数据可统计。
